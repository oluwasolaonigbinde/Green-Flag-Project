# Legacy Business Capability & Migration Coverage Audit

## 1. Executive verdict

Verdict: FAIL

The completed backend is healthy as a modern episode-first replacement architecture, and it implements many workflows the legacy schema only implied: scoped RBAC, append-only audit, Mystery redaction, allocation, scoring, decisions, notifications, messages, exports, and public-map event boundaries. It cannot yet be considered safe to replace the legacy Green Flag Award system from a migration-coverage perspective because core production data has no implemented legacy ID/source-table/import-batch mapping, no import staging, and no reconciliation tooling; additionally, important legacy finance, document, judge-application, contact, public-profile, and resource/content data groups are only partially represented or have no precise migration destination. The KBT_GFA schema appears mostly to be Umbraco CMS/Forms infrastructure and should not drive a CMS rebuild, but form/content/resource data still needs an archive or migration decision where it represents business submissions or retained public content.

## 2. Success criteria checklist

- [ ] PARTIAL - Important legacy business concepts have new backend equivalents. Evidence: episode-first operational concepts exist in `packages/db/migrations/0003_organisations_parks_locations_cycles_episodes.sql` (`assessment_episodes`), S05-S14 migrations, and API tests, but legacy public profile fields, judge applications, document subtypes, finance history, and content/resource records are partial or missing.
- [ ] FAIL - Important legacy data has migration destinations. Evidence: `rg` found no `legacy_id`, `legacy_source_table`, `legacy_import_batch_id`, import staging tables, or reconciliation tooling in `packages/db/migrations`, `apps`, or `packages`; `docs/implementation/production-readiness-checklist.md` lists migration dry run/reconciliation as a future gate.
- [ ] PARTIAL - Important old workflows are represented. Evidence: registration, autosave, documents, submission/payment state, allocation, assessment, decisions, communications, and public-map outbox have endpoints/tests, but judge onboarding/application approval, detailed finance migration, legacy document subtype import, public CMS/resource workflows, and production import verification are incomplete.
- [x] PASS - New backend follows the new episode-first architecture rather than copying legacy shape. Evidence: `packages/db/migrations/0003_organisations_parks_locations_cycles_episodes.sql` defines `award_cycles`, `cycle_windows`, and `assessment_episodes`; `packages/db/migrations/0005_applicant_dashboard_application_draft_autosave.sql` keeps `applications` linked one-to-one to episodes with narrow applicant package status; `packages/contracts/src/contracts.test.ts` has `keeps the application and episode state models separate`.
- [x] PASS - Known legacy weaknesses are improved. Evidence: `audit_events` and `admin_override_events` append-only triggers in `packages/db/migrations/0002_identity_rbac_audit_foundation.sql`; central Mystery redaction in `apps/api/src/redaction.ts`; normalized document metadata/hash/versioning in `packages/db/migrations/0006_documents_management_plan_upload_link_versioning.sql`; typed lifecycle checks across migrations.
- [ ] PARTIAL - Gaps are either fixed, intentionally deferred, or above-scope. Evidence: external production values are documented in `docs/implementation/external-configuration-register.md`, but legacy migration destinations and import verification are not implemented and are only listed as future launch gates in `docs/implementation/production-readiness-checklist.md`.

## 4. Blockers

- No implemented legacy import reference and reconciliation strategy
  - Legacy concept affected: parks, organisations, contacts, applications, awards/assessments, judges, COI, invoices, documents, scores, feedback, notifications.
  - Evidence: no repo evidence found for `legacy_id`, `legacy_source_table`, `legacy_import_batch_id`, import staging tables, ID mapping tables, row-count/hash reconciliation, or import scripts; `docs/implementation/production-readiness-checklist.md` lists obtaining exports, dry run, reconciliation, idempotency, and rollback as future gates.
  - Why it blocks replacement/migration: core old records cannot be traced to new IDs, reruns cannot be proven idempotent, and business users cannot reconcile production totals.
  - Smallest recommended future implementation goal: add import batch/source-reference tables plus reconciliation reports for Park, Organisation, Contact, ParkAwardApplication, Award, ParkApplicationNote, Judge, JudgeConflictOfInterest, Fee, Invoice, ParkDocument, and EmailLog.

- Legacy finance history has no sufficient migration destination
  - Legacy concept affected: Fee, Invoice, TotalCost, VAT, currency, purchase order, billing/contact snapshots, payment history, Business Central/export status.
  - Evidence: `packages/db/migrations/0007_submission_invoice_po_payment_state.sql` stores `amount_marker = external_value_unavailable`, invoice status, due date, PO/no-PO, and manual mark-paid fields only; no repo evidence found for fee schedules, fee lines, invoice lines, VAT/currency, billing snapshots, payment event history, or legacy Invoice ID mapping.
  - Why it blocks replacement/migration: historical invoices and fee schedules are financial records; replacing the old system without preserving totals, VAT/currency, billing parties, and payment evidence risks data loss and operational/finance failure.
  - Smallest recommended future implementation goal: add finance migration coverage for fee schedules/lines, invoice billing snapshots, invoice amounts/tax/currency, payment events, and legacy Invoice/Fee source references.

- Legacy document and filename-only file data cannot be safely migrated or verified
  - Legacy concept affected: ParkDocument, management plan, constitution, lease, insurance, risk assessment, travel directions, feedback files, heritage/conservation docs, innovation docs, photos, judge CV/application/cover/photo files.
  - Evidence: `packages/db/migrations/0006_documents_management_plan_upload_link_versioning.sql` has normalized `document_assets`, but `document_type` is only `management_plan` or `supporting_document`; no repo evidence found for legacy filename source paths, source table/column, import batch, subtype taxonomy, or assessor-profile document linkage.
  - Why it blocks replacement/migration: filename-only legacy fields need deterministic mapping to object storage keys, hashes, business subtype, visibility, and owner records. Without that, important application, feedback, and judge files may be lost or misclassified.
  - Smallest recommended future implementation goal: add document import mapping from legacy filename fields into document assets/result artifacts/assessment evidence/assessor documents with subtype, source table/column, storage key, hash, and reconciliation checks.

- Core legacy contact and judge identity mapping is incomplete
  - Legacy concept affected: Contact, ParksContact, application contacts, Judge, JudgeApplication, Administrator, User/member records.
  - Evidence: `internal_users`, `role_assignments`, and `assessor_profiles` exist, but no repo evidence found for legacy Contact/ParksContact/Judge/JudgeApplication ID references, park contact tables, organisation contact tables, judge application records, or assessor document/onboarding import.
  - Why it blocks replacement/migration: users, contacts, and assessors are relationship-heavy; losing legacy IDs breaks ownership, allocation history, COI mapping, notification history, and support reconciliation.
  - Smallest recommended future implementation goal: add legacy person/contact import mapping for users, contacts, park contacts, organisation contacts, judges, judge applications, and role/scoped identity links.

## 6. Medium/low-risk gaps

- Country/Region/County/Authority are denormalized to text in locations. This is acceptable architecturally if intentional, but migration needs a reference-to-text mapping and duplicate/canonical-name report.
- EmailLog import is not implemented. Notification queue/logs are a good modern destination, but old delivery body/error/resend history needs archive/import mapping.
- Public map dispatch is only an outbox. `public_map_update_events` is a sound boundary, but production endpoint credentials, worker dispatch, and reconciliation are external.
- Notification templates use `external_template_copy_unavailable`. This is appropriate until approved copy arrives, but legacy template/content migration needs a plan if old templates remain authoritative.
- Public events, votes, contact forms, resource logs, and resource download logs are not operational blockers if product excludes them, but retention/consent/archive decisions are still needed.
- Homepage display and other marketing flags have no modern destination. They should not pollute award lifecycle tables, but should be mapped to a CMS/public-content plan if still used.
- Production storage, virus scanning, signed URL generation, email/SMS providers, Business Central, payment provider automation, official criteria, and approved bands remain external configuration gates. These are documented, but UAT/go-live must not treat lower-env stubs as production-ready.
- Tests cover many dangerous new workflows, but there are no migration/import tests because import tooling does not exist.

## 7. False positives / legacy noise

- Legacy one-row application lifecycle booleans and status integers should not be rebuilt. The new target correctly splits `applications`, `assessment_episodes`, `allocations`, `judge_assessments`, `decision_results`, and `park_award_cache`.
- Legacy filename-only document columns should not be recreated. They should be imported into normalized document/result/evidence assets with storage keys, hashes, visibility, subtype, and source metadata.
- Legacy `ApiAuth`, legacy `User.Password`, `cmsMember.Password`, and Umbraco user passwords should not be migrated as credentials. Re-provision through Cognito/internal users and archive old auth records only if required.
- Umbraco CMS infrastructure tables such as nodes, content types, templates, macros, cache, redirects, webhooks, and relation tables should not be rebuilt in the awards backend. Only actual business submissions/content/resources need archive or migration decisions.
- Research Centre `RC_*` tables are separate legacy noise unless product confirms they are part of the Green Flag Award replacement.
- ParksVote and resource download analytics should not be rebuilt as core award workflow unless product requires analytics continuity. Archive/export is usually enough.
- `IsShowingFlag`, `WonGreenFlagAward`, `IsResultsLive`, and similar booleans should be translated into `decision_results`, `park_award_cache`, and `public_map_update_events`, not copied as authoritative flags.
- Production fee/VAT/scoring/band/legal/provider values are intentionally not hardcoded. This is good; the gap is not the absence of invented values, but the absence of migration-safe finance/config destinations for legacy historical data.
